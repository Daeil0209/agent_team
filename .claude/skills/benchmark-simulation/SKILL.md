## External-Reference Methodology
When benchmarking against external references (web research, public examples, reference UIs, industry best practices, standards), the methodology MUST be **abstraction-fit-adapt**, NOT direct copy. Direct copy transplants source-context defaults into destination context that often differs in user, scale, constraint, or domain — producing misfit dressed as best-practice.

Required cycle for each external reference:
1. **Abstract**: identify the design problem the reference solves, the underlying principle that makes it work, the user need it serves. Strip away surface details.
2. **Fit-judge**: evaluate against destination (does our user have the same need? same scale? same constraint? same domain conventions? same regulatory context?). Reject patterns where fit fails.
3. **Adapt-synthesize**: combine fitting abstract patterns into a NEW design proposal that solves OUR specific problem. The output is a design synthesis informed by external evidence, not a list of borrowed elements.
4. **Reject misfits**: explicitly record patterns considered but rejected with fit-failure reason. Silence on rejection invites later reintroduction by drift.

**Mechanical skill-load** (loading benchmark-simulation + enumerating external patterns + recommending direct adoption) is beginner-level use of this skill and constitutes a methodology defect. The skill's value is in the abstraction-fit-adapt cycle, not in the external-pattern enumeration. External evidence ALONE without fit-judgment widened the search but didn't solve the destination problem.

### Reference-Class Discrimination
External references split into two structurally distinct classes that require different citation discipline (grounded by Round-N external benchmark of major design systems Material/Carbon/HIG/Polaris vs. W3C/WCAG/ISO accessibility specs):
- **Standard-class references** (W3C, WCAG, ISO, ANSI, regulatory body specs, framework-official testing guides) carry numeric compliance contracts and explicit thresholds — cite literally with tier-1 authority. They are evidence at face.
- **Principle-class references** (Material/Carbon/HIG/Polaris design principles, Nielsen Norman heuristics, design-system identity declarations) carry synthesized methodology — they must enter the abstraction-fit-adapt cycle as input, NOT be quoted as authority. They are starting material for synthesis, not closing evidence.

Citing a principle-class reference as if it were a standard-class is a methodology defect (the source's identity declaration becomes mistaken for empirical proof). Citing a standard-class reference as if it were a principle (inviting re-interpretation of a numeric threshold) is the opposite defect.

When a source claims research-grounding ("rigorous research", "user-driven", "evidence-based") but does NOT link to the underlying study, sample, or findings, downgrade from `evidence` to `assertion` and lower one tier on the source quality ladder — research-artifact accessibility is the discriminator between research-class and principle-class.

### Multi-Modal Observation Discipline
External evidence is rarely text-only. The web is rich with multi-modal information; collecting only the text excerpt leaves most evidence on the table — and for visual / structural / UX decisions, text-only is itself a fidelity defect ("frog-in-well" reasoning bounded to model prior knowledge).

Required modalities per reference (use what the source supplies; see `references/observation-protocol.md` for per-modality discipline):
- **Text**: design rationale, principle articulation, user-research findings, anti-pattern callouts
- **Image / screenshot**: full layout capture, isolated component capture, before/after pairs, density-state captures
- **Quantitative measurement**: font-size ratio (heading:body, primary:secondary), image-to-text ratio, whitespace ratio, color count, grid module count, action-density-per-region
- **Structural form**: table layout (column count, header treatment, alignment, separator), grid system, navigation pattern, form-flow pattern
- **Color, spacing, typography**: color semantic mapping (alert / success / neutral), spacing rhythm (4 / 8 / 12 / 16 base), typographic scale
- **Interactive / behavioral**: animation, transition, state change (infer motion from documented behavior or demo videos when source is static)

A reference observation is incomplete if it documents only one modality on a multi-modal decision. Downstream consumers cannot make a fit-judgment without the modalities the destination depends on.

Each iteration enriches the team's design vocabulary and adaptive fit-judgment, which is the mechanism by which design capability evolves continuously rather than plateauing.

### Verification Methodology Composition
The abstraction-fit-adapt cycle applies to **verification methodology** itself, not just to design surfaces. When choosing how to verify (test framework selection, proof tooling, coverage strategy, correction-criteria design), benchmark external best-practice the same way:
- compare verification patterns from authoritative sources (Google Testing Blog, Martin Fowler, Kent Beck, ThoughtWorks Tech Radar, framework-official testing guides)
- assess fit (does our user-surface match? does our scale match? does our risk profile match?)
- synthesize verification approach from fitting patterns + reject misfits
- compose with `tester` lane for execution and `validator` lane for final acceptance

Verification-methodology benchmarking is mandatory when the verification approach materially affects acceptance quality AND prior session has not already frozen the verification basis from external evidence. See `references/verification-methodology.md` for protocol.

**Result-level verification mandate**: per CLAUDE.md `[USER-SURFACE]`, design output must be verified at the rendered/operating result level (real browser interaction, real reader/operator surface), NOT source markup. The composition's value is realized only when the rendered/operating deliverable matches the design synthesis intent; `dev-workflow` gap iteration operates on result-level diff, not source-level diff. Source-only verification when result-level is feasible is itself a verification defect for visual-UI deliverables.

When dispatching this skill, lead should pre-name which sibling skills the receiving lane should compose with based on deliverable phase per the Smart Composition Recipe above.

Use `BENCHMARK-MODE: light|full` exactly as frozen in the packet. Benchmark-light is the default comparative discipline; benchmark-full is required when routing, promotion, or sustainable team shape materially depends on the result.

## Capability-Evolution Loop
This skill exists to compound the team's capability over time. Each cycle deposits durable artifacts into the team's evolving design / methodology vocabulary:
- **Pattern catalog (positive)**: patterns proven to fit the destination context — preserved for future similar problems
- **Anti-pattern catalog (negative)**: patterns considered and rejected with explicit fit-failure reason — prevents drift re-introduction in future cycles
- **Strength preservation memory**: destination strengths the iteration must NOT lose — protects against iteration regression
- **Confidence-rated synthesis claims**: high (multi-source authoritative + strong fit) / medium (one source OR partial fit) / low (single source + weak fit OR speculation)
- **Reference provenance ledger**: source URL + capture date + claimed pattern per reference — uncited memory is `HOLD` not `evidence`

A benchmark cycle that does not deposit any of these is a wasted cycle (got the answer this round but did not get smarter). A cycle that pollutes the catalogs (uncited claims, untested fit, copy-paste patterns) is worse than wasted (it actively degrades future judgment).

### Pattern Lifecycle Staging
Per ThoughtWorks Technology Radar discipline (Adopt / Trial / Assess / Hold), each pattern catalog entry carries an explicit lifecycle stage to prevent stale-pattern accumulation:
- **Assess** — observed in external evidence, not yet synthesized into our doctrine
- **Trial** — synthesized into our doctrine and applied to at least one destination with feedback captured
- **Adopt** — verified stable across 2+ destinations and stable across iteration cycles
- **Hold** — formerly considered (or formerly Trial/Adopt) and now actively-recommended-against; promoted from reject log to active warning when destination application revealed fit failure

Hold is **active warning, not silent removal** — when a previously-considered pattern proves fit-failed, record it in Hold with the fit-failure reason; do not silently delete. Silent removal of catalog entries is a discipline failure (the team loses the lesson).

Stale `Assess` or `Trial` entries surface on each re-invocation of benchmark-simulation for promote-or-fade decision. Indefinite `Assess` status is not allowed — every benchmark cycle either promotes or demotes its predecessors' Assess/Trial entries. Catalogs that grow monotonically without periodic prune-and-promote review degrade into noise rather than evidence.

### Axis-Coverage Discipline
Per converged external evidence (NN/g 3-axis research framework, Google SMURF 5-axis verification, Fowler 3-axis testing-style, ISO 25010 2-axis quality), methodology choice is structurally **multi-axis, not one-dimensional**. Single-axis selection (e.g., "pick the fastest test", "pick the most-cited reference", "observe only the text modality") under-specifies and produces brittle methodology choices.

Each benchmark cycle must record an explicit axis-coverage check at closure: which axes were exercised in this cycle? which were skipped, and why? Skipping an axis silently is the multi-axis equivalent of single-modality observation — incomplete by structural definition.

Closure standard: mark `decision-ready` only when (a) synthesis is grounded in cited multi-modal evidence per `External-Reference Methodology`, (b) at least one catalog above has a new entry from this cycle (with explicit lifecycle stage), (c) the reject log is non-empty (silence on rejection means fit-judgment was not actually exercised), AND (d) axis-coverage check is explicit (which axes exercised, which skipped + why).
