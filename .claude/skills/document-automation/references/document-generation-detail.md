# Reference Material
### Proposal/Plan/Grant
- Separate explicit requirements from inferred evaluation intent; keep both visible.
- Require organizational strategy capture: current situation, strategic motivation, focus item, roadmap position, desired outcome type; flag absence as intake gap.
- Design full logic and section dependency first, then compress to target page/length; do not pre-allocate fixed page counts per section.
- Generation order: requirement reading → evaluator-intent inference → benchmark-pattern intake → strategy capture → build-vs-emphasis target → logic-line design → drafting → visual selection → page-fit compression.
- For execution-control documents, prefer checklists, schedules, allocation tables, and role maps over long persuasive prose.
- Keep prose in forward-looking execution/design voice; do not let the document read like a retrospective result report.
- Treat fatigue, concentration rhythm, and post-transition performance drops as realism inputs for schedules.
### Research/Survey Reports
- Freeze supported question, primary reader, and decision before intake or drafting.
- Freeze a method-shaped skeleton: question scope, source-family coverage, inclusion/exclusion logic, cross-check state, uncertainty labels, update-trace surfaces.
- Structure outputs in reader decision order: decision summary → judgment-driving analysis → evidence detail → update-ready trace.
- Anchor skeleton to a methodology framework (PRISMA for literature, Cochrane for systematic review, WIPO for patent landscape); declare it before prose fill.
- Classify each cited paper by epistemic role (foundational, review/survey, critical/rebuttal, application/case); imbalance is a literature-character defect.
- Patent analysis requires: CPC/IPC codes, key applicant filtering, patent-family grouping, legal-status filtering, filing-date windowing; absence is a shallow-analysis defect.
- Search strategy must be concept-axis decomposition: identify independent axes, expand with synonyms, combine with Boolean logic; keyword-only search is a defect.
- Required visual palette evaluation: technology taxonomy diagram, comparison matrix, distribution map, competitive positioning map, claim-evidence mapping table.
- Body = analysis advancing the reader's decision; appendices = supporting detail, raw data, extended methodology; body sections reading as appendix material are a structure defect.
### Operational Plans/Schedules
- Preserve hidden time, transition burden, dependency/resource conflicts, buffer logic, approval checkpoints, and version-management expectations.
- Distinguish notice-facing vs. execution-control variants; choose representation (table, timeline, gantt, checklist, split-version) by audience, dependency shape, change rate, and misread risk.
- Generation order: mode/audience → constraints/resources → representation/tooling → arrangement → feasibility review → version split.
- Place buffers by risk segment (post-travel, pre-transition, post-meal, multi-person movement, external-variable); buffer absence in high-risk segments is a feasibility defect.
- Execution-control minimum field set: activity, time, responsible person, location, preparation items, prerequisites, status, approval state.
- Mandatory feasibility review: hidden-time completeness, responsible-person duplication, equipment/resource conflict, buffer sufficiency, overcrowded slots, approval-timing gaps, parallel-task interference; any failure marks `HOLD`.
- Acceptance priority order: (1) executability, (2) zero-ambiguity comprehension, (3) change resilience, (4) hidden-time completeness. Visual tidiness is not an acceptance criterion when it conflicts with these four.
- Cascade structure from purpose type: education-focused schedules prioritize concentration/rest rhythm, event-focused prioritize flow/transition continuity, development-focused prioritize dependency/approval sequence; a multi-purpose schedule without explicit priority ranking among purpose types is a structural defect.
## Review and Analysis Report Persuasion Structure
Apply "Why → So What → Now What" for risk analyses, condition evaluations, and go/no-go reports. For short answer-first condition reviews, compress to: answer → decisive reasons → actionable conditions.
- **Stage 1 — Why:** Decompose into 4–6 distinct named categories each with concrete causal mechanism (e.g., "R-290 flammability range 2.1–9.5 vol%"); category labels alone are a persuasion defect.
- **Stage 2 — So What:** Dedicate a named section to why the target environment amplifies identified risks; generic restatement is a defect.
- **Stage 3 — Now What:** Tier recommendations by risk level (low → medium → high); one-size-fits-all guidance across different risk contexts is a usability defect. Short answer-first artifacts with one dominant decision path may use a single standard path plus bounded exceptions.
- Case studies: include only when materially available, decision-changing, or explicitly requested; not a universal minimum.
- Bidirectional framing: present both what must NOT be done and what conditions MUST be met; unidirectional framing is a completeness defect.
## Information Density Design
- Core conclusion accessible within 30 seconds; place summary conclusion table or executive summary at document top.
- Three-layer reading design: 30-second layer (conclusion), 3-minute layer (section headings + key tables), full-read layer (body). Each layer must be independently coherent.
- Prefer tables over prose when the same information fits either form; apply decision-necessity test per paragraph.
- Explicitly state document scope and limitations at the beginning.
- Apply persuasion depth only to the decisive argument; default all other sections to density-first (tables, 30-second accessibility, decision-necessity filtering).
## Practical Application Tools
- Include judgment-column checklists (e.g., Suitable / Needs revision) when the reader will reuse the document as an execution or field-check tool; short memos may use a compact condition table instead.
- Conclusion should include a numbered implementation sequence when the reader needs an execution path, not just a judgment.
- For operational condition tables, add a "Practical judgment" column; short memos may use a concise interpretation note instead.
- Tier recommendations by risk level; untiered recommendations in risk-differentiated contexts are a usability defect.
## Introduction and Reader Preparation
- Macro-to-micro zoom: start from global/industry trend → specific domain → specific environment; micro-first without macro context is a reader-preparation defect.
- Taxonomy pre-education: explain specialized classification systems (ASHRAE, ISO, etc.) before using them in analysis.
- Self-selection axis: structure key reference tables so one axis represents the reader's self-identified category; absence forces linear reading and is a usability defect.
## Analysis Depth Techniques
- **Causal chain:** Within each risk category, narrate trigger → mechanism → immediate consequence → ultimate outcome; stopping at risk label is an analysis-depth defect.
- **Cascade risks:** After primary risks, check for secondary dangers emerging as consequence; stopping at primary risks is a defect.
- **Sensory limitation:** When risk is invisible (colorless, odorless), state this explicitly to preempt "I'll just be careful" objection; omission is a persuasion gap.
- **Regulation-reality gap:** When gap data between regulatory requirement and actual practice is available, include it; omission in compliance-review reports is a persuasion defect.
## Conclusion Design Techniques
- State an unambiguous decision criterion ("if X, Y, Z not met, the activity should NOT proceed"); vague advice is a judgment-delivery failure.
- When competing priorities exist, explicitly declare which takes precedence.
- When reader's situation is undetermined, state a conservative-design principle (judge by the most hazardous possible condition); omission is a judgment-delivery defect.
- For legal/regulatory tables, add a column translating each item into the reader's specific context; absence is a usability defect.
- Optimal length test: removing anything loses a critical argument; adding anything dilutes focus. Apply impact-per-page test before finalizing.
## Event Planning Document Generation
Use when document mode is institutional event planning (retreats, workshops, ceremonies, team-building, training camps).
### Pre-Generation Gate
Before generation, freeze:
- Purpose type: education-focused, cohesion-focused, performance-sharing, morale, or hybrid
- Target participant response; approval-path logic and format constraints
- Atmosphere/mood type (calm exchange, light relaxation, active socializing, seminar-included, hybrid) as a separate intake axis from purpose type
- When participant power is asymmetric, senior authority preferences are hard constraints; general participant preferences are soft constraints resolved after authority constraints are satisfied
- Do not begin element selection or structure drafting until purpose type, target response, and approval path are frozen
### Element-to-Region Selection
- Standard element palette: education sessions, meals/catering, recognition/awards, networking/social, free time/rest, logistics/transport, opening/closing ceremonies, team activities, guest speakers, breakout sessions
- Element entity model (definitions, constraints, institutional norms) owned by `business-workflow`; consume as input.
- Each element requires a purpose-link justification for inclusion; exclusion requires brief rationale; omit region when purpose does not justify the element.
- When transport mode is personal vehicles, promote to an independent major section with a structured vehicle-allocation table (driver, passengers, route, departure time).
- When mandatory elements impose venue-level physical requirements, use operational-feasibility criteria (equipment, space layout, fire safety, weather exposure), not aesthetic language.
### Generation Stage Sequence
1. **Purpose/audience/constraints intake** — freeze purpose type, participant profile, venue constraints, budget envelope, institutional culture norms, approval path
2. **Full element enumeration with necessity checks** — apply `why needed / removable / simpler alternative` check per element
3. **Logical dependency and sequence design** — arrange elements respecting temporal dependencies, participant energy curve, logistics transitions, venue constraints; preserve rationale chain for ordering decisions
4. **Compression preserving rationale chain** — fit to target length without silently dropping necessity justifications
Between stages 1 and 2: classify venue type (retreat/relaxation, gathering facility, urban facility, outdoor/camping) and cascade into preset adjustments to schedule density, free-time allocation, collective-vs-individual time ratio, and meal operation mode.
### Participant Burden Constraints
- Treat fatigue, concentration rhythm, logistics complexity, transition overhead, and post-meal energy as hard realism constraints.
- Conservative defaults when thresholds not provided: max 90-min sessions before break, min 15-min transitions, lighter content after meals, progressive intensity reduction across multi-day events.
- Schedules violating these constraints: `HOLD`.
### Visual Inclusion for Event Documents
- Candidate types: schedule/timetable tables, process flow diagrams, venue layout maps, role/responsibility matrices, budget allocation charts, participant journey maps.
- Apply semantic duplication check before including any visual; rendered composition quality belongs to `visual-composition`.
### Tone and Repeatability
- Formal enough for approval circulation but not so rigid that operational instructions become ambiguous; rationale-first over embellishment.
- Before finalizing: (1) Is the structure reusable for the next similar event without broad rewrite? (2) Is operational complexity within the capacity of the practitioner executing it? If either fails, simplify or add execution guidance.
## Result Report Generation
Use when generating a result report from a plan, proposal, or project reference.
### Adaptive Generation Levels (L1–L3)
- L1 (rich plan): summarize and reorganize, minimize inference.
- L2 (standard plan, default): generate domain-appropriate process narrative filling structural gaps.
- L3 (sparse/outline-only): proactively supply expert-level structure and bridging content within evidence-safe bounds.
- Assess generation level per section, not per document.
### Voice and Evidence Rules
- Default to bulletin-style (gaejosik): short declarative sentences, key information first, no padding.
- Hybrid voice: factual-report tone for body/method sections; performance-emphasis tone for expected-effect and significance sections.
- Experiments, tests, measurements, and validation results must not be written as performed unless actual evidence is provided; plan-described experiments are referenced only as planned or intended.
### Visual Rules
- PPT: co-locate text with supporting visual per slide; balanced text-to-visual ratio; font sizes legible inside all visuals.
- Word/HWP: images occupy at most ~one-third of page area; every image requires a descriptive caption.
- Short docs (2–3 pages): heavier visual ratio only when visuals compress decision-relevant information better than text. Long docs (10+ pages): ~5:5 to 6:4 text-to-visual ratio.
### Completion Conditions
- Target submission-ready quality where human role is reviewer/approver, not rewriter; fill logical, structural, and transitional gaps within evidence-safe bounds.
- Page-count mismatch is a defect; adjust via text-visualize-compress cycle.
## Role-Scoped Structural Feedback
- Challenge manager instructions, peer handoffs, or generation plans when template fidelity, source-to-output mapping, document-state rules, evidence boundaries, or formatting preservation requirements are under-specified.
- State the issue in document-automation language: which template region, mapping rule, generation stage, reader-job contract, or fidelity constraint is weak, and the smallest corrective rewrite.
- Do not silently automate a document path whose structure is too vague to preserve operator-facing quality; silence is a lane failure.
- **Challenge routing:** structure disputes → `team-lead`; evidence boundary disputes → `validator`; visual composition disputes → `visual-composition`; domain rule disputes → `business-workflow`
